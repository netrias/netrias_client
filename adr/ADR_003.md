# ADR 003: Boundary Validation for Discovery and Manifest Ingress

## Status
Accepted

## Context
The SDK uses array index as the canonical `column_id` binding between CSV columns, the discovery request, the discovery response, and the manifest written to disk. Two weaknesses in the ingress boundary left this invariant vulnerable:

1. **Discovery response trusted on length alone.** `_suggestion_from_entry` used the array index as `column_id` and only cross-checked that `len(results) == column_count`. A backend regression that sorts results alphabetically (or any other reorder) would preserve length but shift every column identity downstream — the SDK would silently bind the wrong CDE to every source column. This is exactly the failure mode the position-indexed design exists to prevent.

2. **Manifest ingress validated keys but not value types.** `_coerce_entry` checked that the five required keys were present, then cast the mapping to `ColumnMappingRecord` and returned it. A manifest entry with `cde_id: "forty-two"` or `harmonization: "bogus"` passed validation and produced a TypedDict that lied about its fields. Downstream consumers treating the record as typed would encounter type errors far from the actual point of violation.

Additionally, the runtime mirror of the `Harmonization` literal (`frozenset[str]`) lived in `_discovery.py`, and the repeated `"column_name"` string literal had no module-level constant — both would drift as soon as a second module needed them.

## Decision

### 1. Positional parity cross-check at the discovery boundary
Thread the tuple of outbound `column_name` values through the response parser. At each result index, require `results[i].column_name == columns[i].column_name` exactly; a mismatch raises `MappingDiscoveryError` with the expected and received names and the source index. Length parity alone is no longer trusted to guarantee identity parity.

### 2. Typed transport contract
Change the three transport function signatures (`request_mapping_discovery`, `invoke_cde_recommendation_alias`, `discover_via_step_functions`) from `list[dict[str, object]]` to `list[ColumnSamples]`. The typed wire shape is preserved end-to-end instead of being erased with a `cast` at the module boundary.

### 3. Value-type validation at the manifest boundary
Extend `_coerce_entry` to assert, per required field, that the value matches the declared domain type: `column_name`/`cde_key` are strings; `cde_id` is a non-bool int; `harmonization` is in the allowed set; `alternatives` is a list. Failures raise `MappingValidationError` with the field name, expected type, received type and value, and `column_mappings[i].<field>` source.

### 4. Single source of truth for boundary constants
- `HARMONIZATION_VALUES: frozenset[str]` and `COLUMN_NAME_KEY: Final[str]` live in `_models.py`, derived from the `Harmonization` literal.
- Both the discovery boundary and the manifest boundary consult the same frozenset, and response/manifest lookups reference `COLUMN_NAME_KEY` instead of repeating the raw literal.

### 5. Fail fast on zero-column CSV
`_samples_from_csv` rejects an empty header row with `MappingValidationError` before any request is issued. A degenerate input no longer produces a trivially-"successful" empty manifest.

## Consequences

**Benefits:**
- A backend reorder that preserves length is caught at the boundary instead of corrupting every downstream binding.
- Manifest consumers receive either a record that matches its declared types or a typed error — never a TypedDict that lies about its fields.
- Allowed harmonization values and the column-identity key have one runtime owner; the set and key cannot drift between modules.
- Transport layers carry the typed wire contract, making future shape changes a compile-time change set.

**Trade-offs:**
- Stricter boundaries convert some tolerable payloads (e.g., a manifest with an integer `column_name`) into hard failures. This is deliberate: silent coercion of mistyped data was the failure mode.
- A backend regression that reorders results or drops a field will surface as `MappingDiscoveryError` rather than being absorbed. This is the intended behavior; position-indexed outputs require position-indexed inputs.

**Migration notes:**
- No public API changes. All new failure modes surface through existing `MappingDiscoveryError` / `MappingValidationError` exception types.
